### Rails console
Can be accessed through `rails console` or bin/rails

### Routing in Rails

CRUD records in rails routing can be simplified into `resources :<name of model>`

For individual routing, look at the code below
```
get "/<name of model>", to: "<name of model>#index"

get "/<name of model>/new", to: "<name of model>#new"
post "/<name of model>", to: "<name of model>#create"

get "/<name of model>/:id", to: "<name of model>#show"

get "/<name of model>/:id/edit", to: "<name of model>#edit"
patch "/<name of model>/:id", to: "<name of model>#update"
put "/<name of model>/:id", to: "<name of model>#update"

delete "/<name of model>/:id", to: "<name of model>#destroy"
```

### Controllers and Routes
Controllers handle the function logic when the route is accessed.
This is similar to Django view.

### Going into controllers and views
Views are Ruby version of templates(In Django).
The linking of views in controllers is through the function name
```
Example:
class TasksController < ApplicationController
  def index
    @tasks = Task.all
  end

  def show
    @task = Task.find(params[:id])
  end
end

and in views/tasks, there should be 2 files
index.html.erb and show.html.erb
This will allow Ruby to reference when accessing
```

`<name of model>_path` will lead to the show function of the controller.

`new_<name of model>_path` will lead to the new function of the controller.

Ruby has strong parameters which helps to add a form of defense against any potential attacks.
